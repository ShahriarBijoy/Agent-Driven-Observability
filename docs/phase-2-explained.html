AI Observability Lab · Phase 2

Application Observability, explained simply

We taught our app to tell us what it's doing — every request, every number, every step — and built a control room where we can watch it all. Here's the whole idea, with no jargon left unexplained.

01The problem we're solving

In Phase 1 we built a working app: an AI gateway that takes a question, looks up relevant text (this lookup-then-answer pattern is called RAG — Retrieval-Augmented Generation), and returns an answer. It's made of several small services talking to each other.

That app works — but it's a black box. If it gets slow, or starts throwing errors at 2 in the morning, how would you even know? And if you knew, how would you find out which of the services is the culprit, and why?

Think of it like… A hospital patient. Before Phase 2, the patient can't speak — you only know something's wrong when they collapse. Phase 2 hooks them up to the monitors: heart-rate, temperature, a chart of everything that happened. Now you can see trouble coming, find the cause, and treat it. That's observability: making a system explain itself from the outside.

So Phase 2 didn't add features for users. It gave every service a nervous system that constantly reports its health — and a dashboard (Grafana) where we can read those reports.

02The three kinds of signals

There are three classic types of telemetry — people call them the “three pillars” of observability. You need all three because each answers a different question. Here they are, simplest first:

📝

Logs

THE DIARY

Timestamped text messages about individual events. “2:01pm — chat completed for tenant acme, used 3 chunks.” Great for the story and the details of one thing that happened.

📊

Metrics

THE GAUGES

Numbers measured over time. “120 requests/sec, average 0.3s, 2% errors.” Cheap to store, perfect for dashboards, trends, and alerts — answering “is it healthy right now?”

🧵

Traces

THE JOURNEY

The full path of one single request as it hops between services, with timing for each step. Like a parcel's tracking history. Perfect for finding where something is slow or broke.

When do you reach for which?

You want to know… Use Why
“Is the whole system healthy? Are we slow? How busy?” Metrics Cheap numbers, great for graphs & alarms over time.
“This one request was slow — which step ate the time?” Traces Shows every step of one request, end to end, with durations.
“What exactly went wrong? Show me the error message & context.” Logs The detailed, human-readable record of a specific event.
The combo move In real life you use them together, in this order: Metrics shout “something is wrong!” (an alert fires) → Traces point at the guilty step (“the retriever is taking 4 seconds”) → Logs tell you why (“database connection timed out”). Phase 2 is wired so you can hop between them in one or two clicks.

03The tools — who does what

A bunch of pieces have to cooperate to make this happen. Don't be scared by the names — each one has exactly one job. Read them as a relay team.

The language: OpenTelemetry (“OTel”)

OpenTelemetry
The standard

A vendor-neutral standard for describing logs, metrics, and traces — plus the libraries that produce and ship them. Think of it as the USB-C of observability: instrument your code once in this common language, and you can plug into any backend. We never want to be locked to one vendor, so everything speaks OTel.

The collector: Grafana Alloy

Alloy
The post office

Every service ships its telemetry to one address — Alloy. Alloy receives it, batches it, and fans it out to the right storage backend (logs here, traces there, metrics over there). Services never need to know where anything is stored; they just hand the mail to the post office.

The storage + viewer: the “LGTM” stack

These four are all made by Grafana Labs, and their initials spell LGTM (yes, the “looks good to me” joke is on purpose). Three of them store one pillar each; the fourth is the screen you actually look at.

Loki
Logs store

L — stores and searches your logs.

Grafana
The dashboards

G — the web UI / control room. Dashboards, graphs, the service map, the click-around experience. This is the one humans open in a browser (at localhost:3001 in our lab).

Tempo
Traces store

T — stores and serves your traces.

Mimir
Metrics store

M — stores your metrics as time-series (it speaks Prometheus, the standard metrics query language).

Our own little toolbox: @obs/telemetry

@obs/telemetry
Shared helper

A small package we wrote so every service produces telemetry the same way, with one line of setup. No service touches raw OpenTelemetry directly — they all use these friendly helpers. (More on this in §6.)

04How it all connects

Put the relay team in order and the whole pipeline looks like this:

your services one address storage you (gateway, embedder, retriever, model-proxy) ┌──────────────┐ │ 📝 logs │──┐ │ 📊 metrics │──┤ OTLP* ┌─────────┐ ┌── Loki (logs) ──┐ │ 🧵 traces │──┼──────────▶ │ Alloy │──▶├── Tempo (traces) ──┼──▶ Grafana └──────────────┘ │ └─────────┘ └── Mimir (metrics)──┘ (your browser, ▲ │ the post localhost:3001) │ │ office speaks OTel ──────┘ via @obs/telemetry * OTLP = OpenTelemetry Protocol, the standard wire format. Everyone ships to http://alloy:4318

In words:

  1. Each service is instrumented (it produces signals) using our helpers.
  2. It ships everything in the OTLP format to one place: Alloy.
  3. Alloy fans out — logs→Loki, traces→Tempo, metrics→Mimir.
  4. You open Grafana, which reads from all three and shows it on one screen.

05A tiny worked example

Let's follow one chat request — someone asks the gateway “what is pride and prejudice about?” — and see what each pillar captures.

The request fans out through several services. The trace records that shape and the time spent in each step: 

# A TRACE — one request, broken into timed "spans" (steps)
rag.chat                                    312ms   ◀── the whole request
├─ rag.retrieve                             190ms
│   ├─ POST embedder   (turn text → numbers) 70ms
│   └─ POST retriever  (find top-3 chunks)  115ms
├─ rag.augment         (stitch context)       2ms
└─ rag.generate                             118ms
    └─ POST model-proxy (write the answer)  116ms

Instantly you can see the answer-writing and the chunk lookup dominate the time. That nested picture is exactly what Tempo stores and Grafana draws.

Meanwhile, metrics quietly tick up some counters and gauges:

# METRICS — just numbers, aggregated over all requests
request_duration_seconds{service="gateway", http_status_code="200"}  ← 0.312
tokens_in_total{tenant="acme", model="mock-llm"}                     += 42
tokens_out_total{tenant="acme", model="mock-llm"}                    += 88
cache_hit_ratio                                                       ← 0   (was a miss)
retrieval_relevance_score                                             ← 0.81, 0.77, 0.64

And a single log line records the human-readable outcome:

# A LOG — note it carries the trace_id, so it links back to the trace above
{ "level":"info", "service":"gateway", "msg":"chat completed",
  "tenant":"acme", "topK":3, "model":"mock-llm", "cached":false,
  "trace_id":"4bf92f3577b34da6...", "span_id":"00f067aa0ba902b7" }
The magic link: exemplars See that trace_id stamped on the log, and notice the metric chart has dots on it? Those dots are exemplars — each is a metric data point that remembers “the request that produced me is trace #4bf92…”. So in Grafana you can spot a slow spike on a graph → click the dot → land on that exact trace → click again → read its logs. Metrics → traces → logs, the whole loop, in three clicks. This loop is the headline feature of Phase 2.

06How we actually wired it in code

The whole point of the @obs/telemetry package is that adding observability to a service is almost free. Here's the entire recipe.

1

Turn the engine on — one file, the very first thing the service runs: 

// src/platform/telemetry.ts
import { initTelemetry } from "@obs/telemetry";
initTelemetry({ name: "embedder", version: "0.0.0" });

Then in main.ts it's the first import — so OpenTelemetry is ready before any other code runs: 

import "./platform/telemetry"; // must be first
2

Get traces + RED metrics for every route, for free — mount one piece of middleware (code that wraps every incoming request): 

app.use("*", honoTelemetry("gateway")); // one line per service

This single line opens a trace span for each request and records the fleet-wide RED metrics automatically. RED is an industry mnemonic for the three numbers you always want about a service:

  • Rate — how many requests per second.
  • Errors — how many of them failed.
  • Duration — how long they took.
3

Name the interesting steps — wrap business logic in withSpan(...) so it shows up as a labelled step in the trace (this is the real gateway RAG code): 

return withSpan("rag.chat", async () => {
  const { embedded, chunks } = await withSpan("rag.retrieve", async (span) => {
    span.setAttributes({ tenant: input.tenant, "rag.top_k": input.topK });
    const result = await deps.embedder.embed(input.prompt);
    return { embedded: result, chunks: await deps.retriever.retrieve(...) };
  });
  const completion = await withSpan("rag.generate", () => deps.model.complete(...));
  // ...attach attributes, record token metrics, log the outcome...
});
4

Keep the trace alive across services — when the gateway calls another service, it uses tracedFetch instead of plain fetch. It quietly attaches a “you're part of trace #4bf92…” header, so the next service joins the same trace. This is what draws the Service Map — the diagram of gateway → embedder / retriever / model-proxy.

5

Custom business metrics + structured logs — the gateway also counts things we care about (tokens used, cache hits, rate-limit rejections, relevance scores) and writes JSON logs that carry the trace_id: 

const log = createLogger("gateway");
recordTokensIn(completion.usage.promptTokens, { tenant, model });
log.info("chat completed", { tenant, model, cached }); // trace_id auto-attached
Why we wrote it by hand Normally OpenTelemetry can auto-instrument popular frameworks with zero code. But our services run on Bun (a fast JavaScript runtime), and the auto-magic is built for Node.js — it doesn't hook into Bun. So we built thin, explicit helpers in @obs/telemetry instead. Slightly more code, but it works reliably and keeps every service consistent. (The full reasoning lives in docs/adr/003-application-observability.md.)

07What you can see in Grafana

Once traffic is flowing, Phase 2 gives you these views out of the box:

View What it shows you
Gateway · RED dashboard The health vitals: request rate, error rate, and p50/p95/p99 latency — with clickable exemplar dots.
RAG Pipeline dashboard Business view: token throughput, cache-hit ratio, retrieval-relevance heatmap, rate-limit rejections, top tenants.
Service Map An auto-drawn diagram of which service calls which, built from the distributed traces.
Exemplar → trace → logs The three-click drill-down from a metric spike all the way to the exact log lines.
2 alert rules Auto-fire if the gateway's 5xx error rate > 2% (2 min) or p95 latency > 2s (5 min).

08See it yourself

Two commands and a browser tab:

# 1. Start the app + the whole observability stack
docker compose -f infra/compose.yml -f infra/compose.observability.yml up -d --build

# 2. Pour in some synthetic traffic so the dashboards light up
docker compose -f infra/compose.yml -f infra/compose.observability.yml \
  --profile load up -d load-generator

Then open http://localhost:3001 (Grafana, logs you in anonymously). Open the Gateway · RED dashboard, click a latency exemplar dot to jump to its Tempo trace, then “Logs for this span” to land on the matching log in Loki. That little journey is Phase 2.

09One-screen recap

What Phase 2 did: made every service explain itself, and gave us a control room to watch.

  • 3 pillars — Logs (the diary), Metrics (the gauges), Traces (the journey).
  • OpenTelemetry is the common language all services speak.
  • Alloy is the one inbox; it fans out to Loki / Tempo / Mimir (= LGTM storage).
  • Grafana is the screen you look at.
  • @obs/telemetry makes instrumenting a service a 1–2 line job.
  • Exemplars stitch the three pillars together — spike → trace → logs in a few clicks.

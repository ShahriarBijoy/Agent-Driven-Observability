AI Observability Lab · Phase 6

Chaos, SLOs & Polish, explained simply

We spent five phases building a system that watches everything. Phase 6 asks the only question that really matters: when something breaks, does any of that watching actually help? So we break it — on purpose, on a schedule — and make the system notice, alert, investigate, and explain itself. Here's the whole idea, jargon unpacked, with a worked example.

01The problem we're solving

By the end of Phase 5 the lab was impressive: dashboards, traces, data lineage, and even AI agents that read the telemetry. But there was a quiet, awkward truth hanging over all of it — we had never actually seen a single dashboard turn red. Everything was green because nothing had ever really gone wrong.

That's a trap. A smoke detector you've never tested isn't "working" — it's unknown. An alert nobody has ever watched fire is a guess. A runbook that's never been followed is fiction. Until you break the system on purpose, you don't have observability; you have decoration.

Think of it like… A new office building. It has smoke detectors, sprinklers, fire exits, and an evacuation plan printed on every door. Are they any good? You have absolutely no idea — until you run a fire drill. Phase 6 is the fire drill: we start a controlled fire, and then we watch whether the alarms sound, the right people get paged, and everyone finds the exit. Nobody actually gets hurt, and afterwards you know.

Phase 6 adds five capabilities, each answering a very practical question:

🔥

Chaos

"CAN WE BREAK IT ON DEMAND?"

A scheduler that causes real, repeatable failures on a clock — slow-downs, error bursts, an outage.

🎯

SLOs & alerts

"HOW MUCH BROKEN IS TOO MUCH?"

Written-down reliability targets, and smart alarms that page you when you're burning through them too fast.

💰

Cost control

"CAN WE AFFORD ALL THIS DATA?"

Trimming the flood of telemetry down to what matters, without going blind.

🔬

Profiling & RUM

"WHY IS IT SLOW, AND WHAT DOES THE USER SEE?"

A CPU X-ray of the server, plus the browser's own view of speed and errors.

🎬

The demo

One script that causes an incident and produces a written postmortem, end to end.

02Chaos you can schedule

Chaos engineering is a fancy name for a simple idea: deliberately inject failure into a running system to see how it copes. The key word is deliberately — the failures have to be controlled, repeatable, and easy to stop.

Our lab already had a "fault model" baked into the fake LLM (the model-proxy): it randomly fails a small fraction of requests and occasionally has a "bad minute." But that randomness is decided once, at startup, and just ticks along on its own. For a fire drill we need to choose exactly when the fire starts and stops.

The little control switch inside each service

So we gave the two services that own the interesting failure modes a tiny, clearly-labelled "chaos switch" — an admin endpoint called /admin/chaos that's only for the lab (it can be turned off entirely with one environment variable). It lets us flip failure on and off while the service keeps running, no restart required.

🐌

Latency spike

MODEL-PROXY

Tell the model to answer slowly — around 2–3 seconds instead of milliseconds, but still under the timeout. Requests get sluggish, not broken.

💥

Error burst

MODEL-PROXY

Tell the model to fail most calls (return a 500). The gateway can't get an answer, so it returns a 502 — a real server error.

🔌

Dependency outage

RETRIEVER

Tell the search service (retriever) to go completely dark. The gateway can't build an answer without it, so those requests fail too.

Why real failures, not fake numbers We could have faked a red dashboard by writing bad numbers straight into the metrics. But then we'd only be testing the dashboard, not the system. By making the actual services actually fail, the whole chain gets exercised for real — the gateway's error handling, the metrics, the alerts, the agents — exactly as it would in a genuine incident.

The schedule: a fire drill on a timer

A separate program, the chaos scheduler, reads a plain-text timeline and, minute by minute, flips those switches on and off while also sending normal traffic through the gateway the whole time. The timeline lives in a simple YAML file so you can read and edit it without touching code: 

# chaos/full.yaml — the 26-minute drill
phases:
  - name: latency-spike      # minute 7, for 3 minutes
    target: model-proxy
    startSeconds: 420
    durationSeconds: 180
    params: { latencyBaseMs: 2500 }

  - name: error-burst        # minute 13, for 2 minutes
    target: model-proxy
    startSeconds: 780
    durationSeconds: 120
    params: { p500: 0.6 }     # fail 60% of calls

  - name: retriever-outage   # minute 19, for 2 minutes
    target: retriever
    startSeconds: 1140
    durationSeconds: 120
    params: { outage: true }

The scheduler always turns the chaos back off when a phase ends (and again at the very end, just in case) — a service is never accidentally left broken. Run it with bun run chaos:run. There's a shorter chaos/demo.yaml for the 6-minute demo.

03SLOs: turning "is it up?" into "how much is left?"

Once you can break things on demand, you need to decide how much breakage is acceptable. "Is the site up?" is a yes/no question with no room to think. Real teams use a better frame, and it comes with its own three-letter soup — let's unpack it.

Term Plain meaning In our lab
SLI A measurement — the actual number you track. the % of chat requests that succeed
SLO The target for that number — your goal. "99.5% of chat requests should succeed"
Error budget The allowance — how much failure the target leaves room for. the leftover 0.5% you're "allowed" to fail
Think of it like… A monthly mobile data plan. Your SLO is "don't run out of data this month." Your error budget is the gigabytes you're allowed to use. You don't panic at the first megabyte — but if you burn a third of the month's data in one afternoon, that rate is alarming. SLO thinking watches the rate you spend reliability, not just whether you've run out.

We wrote down three SLOs for the lab, each as a small file in slo/:

SLO Target What it measures
Availability 99.5% chat requests that don't return a server error
Latency 95% chat requests that finish in under 1.5 seconds
RAG quality 90% searches whose best match is actually relevant (score ≥ 0.6)
A tiny but important detail: measure exactly To count "requests under 1.5 seconds" precisely, the stopwatch needs a mark at exactly 1.5s — so we added one to the histogram of request times. Same for RAG quality: we added a metric that records the single best match per search (the old one recorded every match). The lesson generalises: choose your measurement buckets to line up with the targets you care about, or your SLO is fuzzy.

04The clever alarm: multi-window burn-rate

Here's where SLOs get genuinely smart. A naïve alarm says "error rate above 2% → page someone." That pages you at 3 a.m. for a two-second blip, and soon everyone ignores it. A burn-rate alarm asks a better question: how fast are we spending the error budget?

And the best version, the multi-window pattern, only fires when the budget is burning too fast over two time windows at once — a short one to react quickly and a long one to confirm it isn't a fluke.

Think of it like… A smoke detector with two timers. If it senses smoke for five seconds it perks up, but it only shrieks if it also still senses smoke over the last hour. A puff of steam from your shower trips the five-second timer but not the hour, so it stays quiet. A real fire trips both — and only then does it wake the house. Two windows = fast and trustworthy.

We set up two flavours per SLO:

🚨

Fast burn → page

Burning 2% of the whole month's budget in one hour. Checked over both a 5-minute and a 1-hour window. This is "wake someone up."

🎫

Slow burn → ticket

Burning 10% of the budget in six hours. Checked over 30-minute and 6-hour windows. This is "look at it during business hours."

When either fires, it flows straight to the Phase-5 incident reporter agent (via the same alert-webhook we built last phase), which investigates and writes a postmortem. The reliability alarm and the AI investigator are wired together.

An honest caveat we chose not to hide This maths only works for strict targets. Our 90% RAG-quality SLO has a huge 10% budget — so huge that "burning it too fast" would require failing more than 100% of requests, which is impossible. Rather than ship an alarm that can never ring, RAG quality simply alerts when it dips below 90%. Knowing when a textbook pattern doesn't apply is half of using it well, and we wrote that down in the design notes instead of pretending.

05Spending less without going blind

Watching everything isn't free. Telemetry storage grows with two things in particular, and we tamed both — importantly, at the collector (Alloy), so the services themselves don't have to change.

Cardinality — the label explosion

A metric's cost is roughly the number of distinct label combinations it has. Attach a customer's user_id to a metric and suddenly one metric becomes millions — one per user. That's called a cardinality explosion, and it's the classic way to accidentally run up a giant bill.

Think of it like… A library. Shelving books by genre (a handful of labels) is cheap and fast. Shelving a separate copy for every reader who ever borrowed it would need a warehouse. So we strip the "one per reader" labels (user_id, request_id) off the metrics before they're stored. The traces and logs keep that detail — they need it — but the metrics stay cheap. As a backstop, we cap how many series any one tenant can create and raise an alarm if it's ever crossed.

Tail sampling — keep only the interesting traces

Storing a trace (the detailed timeline) of every single request is wasteful — most requests are boring and identical. Tail sampling decides after each request finishes whether it's worth keeping. Our rule:

  • keep every error trace,
  • keep every slow trace (1.5 s or longer),
  • and keep a random 10% of the boring rest.

You throw away the noise and keep exactly the traces you'd actually reach for during an incident, and Tempo's storage shrinks dramatically.

The catch — and we wrote it down If you throw away 90% of traces, any count derived from traces is now an undercount. So the numbers that must be exact — the RED dashboards and the SLOs — come from the services' own request counters (which are never sampled), not from traces. The browser metrics in the next section are real counters for the same reason. Every efficiency has a measurement cost; the skill is knowing which numbers you're allowed to approximate.

06Profiling a runtime that hides

Sometimes you know a request is slow but not why. A profile answers that: it's a picture of what the CPU was actually doing — which functions ate the time — usually drawn as a "flame graph." Grafana Pyroscope is the tool that stores and shows them.

The usual way to collect profiles is a small library inside the app. But our services run on Bun, whose engine (JavaScriptCore) the standard profilers simply can't see — they're built for a different engine (V8). So the normal approach is a dead end here.

Think of it like… The runner refuses to wear a fitness tracker, so instead you point a thermal camera at the whole track. eBPF is that camera: it watches from outside the program, down at the operating-system level, so it doesn't care what language or engine the program uses. It profiles Bun just fine.

Because that "camera" needs special permissions on the host machine, we made it opt-in — it never touches the normal stack unless you deliberately turn it on. When it's running, a slow trace in Grafana links straight to the CPU flame graph for that exact service and moment, and there's a dedicated "Profiles" dashboard.

07Watching from the user's chair (RUM)

Every number so far is the server's point of view. But the server can be perfectly happy while the actual person staring at the screen has a miserable time — a slow network, a heavy page, a failed button. RUM (Real User Monitoring) adds the browser's point of view.

The lab already traced every click; Phase 6 makes the browser report proper numbers:

  • Request rate & error rate — how many calls the page made, and how many failed.
  • LCP (Largest Contentful Paint) — a standard "web vital" that measures how long until the main content actually appears on screen. Under 2.5 s is considered good.

These show up in a new "Frontend" row on the gateway dashboard, right beside the server's numbers. And because the browser's request is stitched into the same trace as the server's work, a slow API call is visible in both views at once — the user's slowness and yours are literally the same trace.

08The pieces — who does what

Phase 6 introduces a few new players and re-uses a lot from earlier phases. Read it as one team with a job each.

/admin/chaos
The switch

A dev-only endpoint on the model-proxy and retriever that turns specific failures on and off at runtime.

chaos scheduler
The drill sergeant

Reads the YAML timeline, drives traffic, and flips the switches on a clock (bun run chaos:run).

slo/*.yaml
The targets

The written-down reliability goals — the single source of truth the alarms are built from.

recording rules
Mimir maths

Pre-computed SLO numbers (error rates over each window, 28-day compliance) that the alarms read.

burn-rate alerts
The smoke detectors

Grafana alarms that fire on fast/slow budget burn and route to the incident reporter.

Alloy
reused · now trims

The collector — now also drops high-cardinality labels and tail-samples traces before storing them.

Pyroscope
CPU X-ray

Stores and draws the CPU flame graphs, fed by an opt-in eBPF profiler.

browser RUM
The user's view

The web app reports its own request rate, error rate, and LCP as real metrics.

incident reporter
reused from Phase 5

The AI agent that an alarm wakes up; it investigates the telemetry and writes the postmortem.

09How it all connects

Here is the full loop — from a healthy system to a written postmortem and back to healthy:

minutes 0–6 happy path gateway all green ✓ │ chaos: error burst → model-proxy returns 500s → gateway → 502s │ the SLI moves → availability starts eating its error budget │ burn-rate alert fires (5m AND 1h windows agree) │ Grafana webhook → the agent-service on the host │ the incident reporter queries Loki / Tempo / Mimir to investigate │ a written postmortem lands in the incident inbox │ you ask the RCA assistant a follow-up → it answers from real data │ chaos ends → SLIs recover → the alert resolves itself

Notice how much of this is reused: the metrics pipeline, the traces, the alert webhook, the AI agents — all from earlier phases. Phase 6 didn't rebuild them; it finally gave them a real incident to prove themselves on.

10A tiny worked example

Let's follow one run of the demo, minute by minute, as a story.

1Warm-up. Traffic flows through the gateway; every request succeeds in a few hundred milliseconds. The availability SLI sits at 100%. All dashboards green.

2The fire starts. At the scheduled minute the scheduler POSTs { p500: 0.6 } to the model-proxy's chaos switch. Now 60% of model calls fail. The gateway, unable to get answers, starts returning 502s.

3The numbers move. In Grafana the gateway's error-rate line climbs from ~0% to ~40%. The availability SLI drops below its 99.5% target and starts burning error budget fast.

4The alarm agrees with itself. The 5-minute burn-rate window crosses the threshold quickly; a couple of minutes later the 1-hour window crosses it too. Both agree → the alert fires. (A one-second blip would never have convinced the 1-hour window — that's the point.)

5The AI wakes up. Grafana webhooks the agent-service. The incident reporter spins up, runs real queries against the logs, traces, and metrics, works out that the model-proxy is the culprit, and writes a Markdown postmortem — summary, impact, timeline, evidence, likely cause, recommended actions — into the incident inbox.

6You ask a follow-up. The demo poses a scripted question to the RCA assistant — "what's the failing dependency and why?" — and it streams back an answer grounded in the same telemetry, citing what it looked at.

7Recovery. The burst ends, the scheduler flips chaos off, the gateway recovers, the SLI climbs back to healthy, and the alert resolves itself — no human needed to clear it. The drill is over, and everything worked.

11How we actually wired it in code

A few of the load-bearing pieces, in miniature.

The chaos switch merges over the base config, per request

// model-proxy decides each request against base config + any live override
export function effectiveFaults(base) {
  return { ...base, ...override };   // override is {} until chaos POSTs to it
}

An SLO recording rule (pre-computed for the alarm)

# the 5-minute error ratio for availability
- record: slo:gateway_availability:error_ratio5m
  expr: |
    sum(rate(request_duration_seconds_count{service="gateway",
             http_route="/v1/chat", http_status_code=~"5.."}[5m]))
    / clamp_min(sum(rate(request_duration_seconds_count{service="gateway",
             http_route="/v1/chat"}[5m])), 1)

The "two windows must agree" alert, in one line of query

# 1 only if BOTH the 5m and 1h error ratios exceed the threshold
(slo:gateway_availability:error_ratio5m > bool 0.0672)
  * (slo:gateway_availability:error_ratio1h > bool 0.0672)

The tail-sampling rule

// Alloy: keep errors, keep slow, sample the rest
policy { name = "errors"       type = "status_code"  ... }  // keep all errors
policy { name = "slow"         type = "latency"      ... }  // keep ≥ 1.5s
policy { name = "sample-rest"  type = "probabilistic" ... }  // 10% of the rest

12See it yourself

With the full lab up and the host agent-service running (it needs your local Claude session): 

# the whole story in one command (~6–8 min)
bun run demo:incident

# or drive the full 26-minute chaos cycle by hand
bun run chaos:run

# turn on the (opt-in) CPU profiler
docker compose -f infra/compose.yml -f infra/compose.observability.yml \
  --profile profiling up -d alloy-profiler

Then look around:

  • Grafana (localhost:3001) — watch the gateway error rate climb, the SLO burn-rate alerts fire, the new Frontend row, and the Profiles flame graph.
  • The control plane (localhost:3003) → Incidents — read the postmortem the reporter wrote.
The payoff This is every earlier phase firing at once — instrumentation, correlation, lineage, alerting, and agents — triggered by a problem you caused on purpose. You break it, and the system notices, explains, and recovers, and you can watch the whole story in a few minutes.

13One-screen recap

  • Chaos — a scheduler flips dev-only failure switches on the services on a clock, so incidents are real and repeatable.
  • SLOs — three written-down reliability targets, each with an error budget you're allowed to spend.
  • Burn-rate alerts — smart alarms that fire only when you're spending the budget too fast over two time windows at once, and wake the AI incident reporter.
  • Cost control — drop runaway labels and keep only the interesting traces (all errors, all slow ones, 10% of the rest), while keeping the exact numbers exact.
  • Profiling & RUM — an eBPF CPU X-ray (because Bun hides from normal profilers) and the browser's own view of speed and errors.
  • The demo — one command that causes an incident and produces a written postmortem, end to end.

Observability work feels finished the day you can produce an incident on purpose, have the system narrate what happened, and explain it back in your own words. That's Phase 6 — and with it, the lab is complete.

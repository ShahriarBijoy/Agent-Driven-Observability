Phase 6 · Explained simply

Breaking things on purpose

The lab already watches everything. Phase 6 is where it earns its keep: we cause a real incident on a timer, let the system notice, alert, and narrate it — then explain it back.

The one-sentence version. We taught the fake outages to run on a clock, wrote down what "healthy" means as budgets we can spend, made the collector cheaper without going blind, learned to profile a runtime that hides from normal profilers, and wrapped it all in a single script that produces an incident and a written postmortem end to end.

01Chaos you can schedule

A dashboard nobody has ever seen turn red is a dashboard nobody trusts. So we made failure repeatable. A chaos scheduler reads a simple timeline and, minute by minute, tells the subject services to misbehave — then tells them to stop.

Analogy Like a fire drill. The building isn't really burning, but the alarms, the exits, and the people all get exercised exactly as if it were — on a schedule, so you can watch what happens and do it again tomorrow.

The trick is that the misbehaviour has to be real enough to move the numbers the alerts watch. So each service that owns a failure mode got a tiny, clearly-fenced "control-plane" switch (dev/lab only):

🐌

Latency spike

Tell the model-proxy to answer slowly (but under the timeout) — requests get sluggish, not broken.

💥

Error burst

Tell the model-proxy to fail a large share of calls — the gateway turns those into 5xx.

🔌

Dependency outage

Tell the retriever to go dark — the gateway can't complete a request and errors out.

The schedule lives in YAML (chaos/full.yaml is the full 26-minute cycle; chaos/demo.yaml is a compressed one for the demo), and the scheduler always turns the chaos back off when it's done — a service is never left broken.

02SLOs: turning "is it up?" into "how much budget is left?"

"Is the site up" is a yes/no question with no room to plan. An SLO (Service Level Objective) reframes it: 99.5% of chat requests should succeed. That leftover 0.5% is an error budget — a real allowance you're allowed to spend.

Analogy A monthly data plan. You don't panic at the first megabyte — you watch the rate you're burning through it. Chew up a third of the month's data in an afternoon and that's worth a phone call.

We wrote down three of these:

SLO Target Meaning
Availability 99.5% chat requests that don't 5xx
Latency 95% chat requests finishing under 1.5s
RAG quality 90% searches whose best match is actually relevant

The clever part is the alert. A naive "error rate > 2%" pages you on every hiccup. A multi-window burn-rate alert only fires when you're burning the budget too fast over two timescales at once — a fast one to react quickly and a slow one to confirm it isn't a blip. Fast burn pages a human; slow burn files a ticket.

An honest caveat This maths only works for strict targets. Our 90% RAG-quality SLO has a big 10% budget — so big that "burning too fast" would require failing more than 100% of requests, which is impossible. Rather than ship a dead alert, RAG quality just alerts when it dips below the target. Knowing when a pattern doesn't apply is half of using it well.

03Spending less without going blind

Observability isn't free — storage grows with two things, and we tamed both at the collector.

Cardinality — the label explosion

A metric's cost is roughly the number of distinct label combinations. Attach a user_id and one metric becomes millions. We strip those unbounded labels off metrics before they land (traces and logs keep them — they need the detail), and set a hard ceiling with an alarm if anything ever crosses it.

Tail sampling — keep the interesting traces

Storing every single trace is wasteful; most are boring and identical. So we decide after each trace finishes: keep every error, keep every slow one (≥ 1.5s), and keep a random 10% of the rest. You lose the noise and keep exactly the traces you'd reach for in an incident.

The catch we wrote down Throwing away traces means any count derived from traces is now an undercount. So the numbers that must be exact — the RED dashboards and the SLOs — come from real counters, not from sampled traces. Every efficiency has a measurement cost; the trick is knowing which.

04Profiling a runtime that hides

A profile answers "what is the CPU actually doing when a request is slow" — a flame graph of where the time went. The usual way is a library inside the app, but ours run on Bun, whose engine (JavaScriptCore) the standard profilers can't see.

Analogy The app won't wear the fitness tracker, so we watch from the outside instead — a thermal camera on the whole room. eBPF profiles at the kernel level, so it doesn't care what language or engine the program uses.

Because that camera needs special permissions on the host, it's opt-in — it never touches the normal stack. When it's on, a slow trace in Grafana links straight to the CPU flame graph for that service and moment.

05Watching from the user's chair

All the numbers so far are the server's view. RUM (Real User Monitoring) adds the browser's: how many requests the page made, how many failed, and LCP — Largest Contentful Paint, i.e. how long until the main content actually appears. A new "Frontend" row on the dashboard shows it, and because the browser's request is stitched into the same trace as the server's, a slow API call shows up in both views at once.

06The whole thing, in one command

bun run demo:incident is the elevator pitch. It runs the entire loop while you watch:

# scripts/demo-incident.sh inject chaos error burst → gateway starts returning 5xx ↓ the burn-rate alert fires → webhook to the agent-service ↓ the incident reporter investigates the telemetry ↓ a written postmortem appears in the incident inbox ↓ you ask the RCA assistant a follow-up — it answers from real data ↓ chaos ends → the SLIs recover → the alert resolves
Why it matters This is the payoff of every earlier phase at once: instrumentation, correlation, lineage, alerting, and agents. You cause a problem on purpose, and the system notices it, explains it, and recovers — and you can watch the whole story in six minutes.

Observability work feels finished the day you can produce an incident on purpose, have the system narrate what happened, and explain it back in your own words. That's Phase 6 — and with it, the lab is complete.

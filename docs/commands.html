AI Observability Lab · Commands

One-liner control via obs (PowerShell wrapper at scripts/obs.ps1). It encodes the network-safe up order so the merged compose command never errors out cold. Just want everything on? Run obs all — containers, both host processes, and a load test, each in its own window.

Enable the obs command (once)

# Add an `obs` function to your PowerShell profile, then reload:
Add-Content $PROFILE 'function obs { & "W:\personal-code\observability-tools\scripts\obs.ps1" @args }'
. $PROFILE

Until then, call it directly: .\scripts\obs.ps1 dev.

Everyday commands

obs all [qps] [secs]Everything in one go — containers + agent-service (:8093) + web (:3003) + load test. Host processes each open in their own window. Defaults 120 qps / 600s.
obs upBring up the full lab containers only (subject + observability + lineage). Alias: obs dev.
obs up --buildSame, rebuilding service images first.
obs downTear the lab down. Add -v to also wipe seeded data + Grafana state.
obs load [qps] [secs]Drive steady synthetic traffic. Defaults: 120 qps for 300s. Shaped variants below.
obs demo [qps] [secs]Full cycle: up → wait healthy → load → down.
obs webStart the web control plane (:3003) in this terminal. Ctrl-C to stop.
obs agentsStart the agent-service (:8093) in this terminal — required for RCA chat.
obs psContainer status. Alias: obs status.
obs logs [service...]Follow logs (optionally for named services).
obs urlsPrint the service address table.
obs smokePhase-1 end-to-end smoke test (needs Git Bash on PATH).
obs hostsPrint the host-process commands (below).

Load tests — traffic shapes

obs load spike [peak] [secs]60s baseline → sudden burst (default 400 qps for 120s) → 180s recovery. Watch: p95 climbs, per-tenant 429s appear, then everything recovers — the agent should call it a traffic surge, not a fault.
prompt The app felt sluggish for a few minutes and some users saw “too many requests” errors, but it looks fine now. Can you check what happened and whether anything is actually broken?
obs load ramp [max]Staircase 50 → 100 → 200 → 300 → 400 qps, 90s per step (stops at max). Watch: the knee where latency starts degrading — the lab’s capacity limit.
prompt We’re expecting a lot more traffic soon. Look at the last half hour of load: at what point did the system start to struggle, and which part gives out first?
obs load soak [qps] [secs]Long, moderate soak (default 60 qps for 30 min). Watch: slow drifts — memory, queue depth, anything that only shows up over time.
prompt Traffic has been steady for the last half hour. Give me a health check: is anything slowly degrading that we should catch before it becomes an incident?
obs load drift [secs]Long-prompt-heavy mix (long:80, happy:20) shifts the prompt-length distribution. Should fire: the DQ prompt-drift alert (KS > 0.4 sustained 10m) — a data incident, no service errors at all. Default 1200s; let it run.
prompt A data-quality alert fired, but the service dashboards all look green — no errors, normal latency. What is the alert about, what actually changed, and should I care?
obs load abuse [secs]Abuser-tenant-heavy mix (abusive:70, happy:30) → per-tenant 429 storm. Nothing should page — the agent should conclude “one tenant hitting its rate limit”, not a service fault. Default 300s.
prompt Support says some users keep hitting rate-limit errors. Is the platform having a problem, or is something else going on? Who exactly is affected?

The prompt line under each command is what you paste into the RCA chat during (or right after) the run — click it to copy. Custom mixes work too: set SCENARIO_WEIGHTS (e.g. "broken:60,happy:40" for a 422 storm) before obs load. Scenarios: happy, repeat, long, abusive, broken.

Fail tests — chaos injection

Each obs fail run drives baseline traffic (default 40 qps) the whole time, injects one failure mode on a clock, then cools down; chaos is always cleared on exit (even Ctrl-C). Keep obs agents running so the paged alert webhooks the incident reporter and a postmortem lands in the inbox — then grill the RCA chat on what happened. Click a scenario’s prompt to copy it, then paste it into the RCA chat while the drill runs (or just after) — each is written like an on-call engineer would ask, without giving the answer away.

obs fail latencymodel-proxy slows (2.5–6s) but never times out — zero errors. Pages: p95 > 2s + latency SLO burn. Pinpoint: the slow model-proxy span in traces. ~14 min.
prompt Users are complaining that chat suddenly takes forever, but I don’t see any errors anywhere. Can you find out where the time is going and what changed?
obs fail errorsmodel-proxy 500s (60%) → gateway 502s. Pages: 5xx > 2%. Pinpoint: 502s trace to model-proxy 500s; retriever/embedder healthy. ~8 min.
prompt The 5xx alert just fired and requests are failing. What’s the root cause? Walk me through the telemetry that proves it.
obs fail timeoutmodel-proxy stalls 15% of calls past the gateway’s 8s upstream timeout. Pages: 5xx > 2% — but they’re 504s, and failing spans cut off at exactly ~8s. A hang, not an error. ~9 min.
prompt We’re seeing a spike in failed requests. Can you figure out exactly how they’re failing and which component is responsible?
obs fail outageretriever hard-down (503 on every call) → ~every request 502s. Pinpoint: the failing dependency is the retriever, not the model. ~8 min.
prompt Everything is failing at once — basically every chat request errors out. What’s down, how big is the blast radius, and what’s the root cause?
obs fail brownoutretriever fails only ~10% of calls — dashboards look “mostly fine” while the error budget burns. Pinpoint: intermittent retriever 503s in only some traces. ~10 min.
prompt Most requests are fine, but users keep reporting occasional errors and the error budget is burning. Is there a real problem hiding in there? Dig into it.
obs fail flakymodel-proxy “bad minutes”: 45s bursts of ~60% 500s every couple of minutes, healthy in between. Pages: 5xx alert likely flaps. Hard mode — the agent must spot the on/off time pattern. ~14 min.
prompt We keep getting paged and then it resolves itself a few minutes later, over and over. Is this a real issue or a noisy alert? Find the pattern and the cause.
obs fail throttlemodel-proxy sheds 50% with 429s → gateway propagates model_overloaded. 4xx, so nothing pages — a deliberate negative control: can the agent find the story with no alert pointing at it? ~8 min.
prompt Nothing is paging, but users are reporting “model overloaded” errors. Why is that happening, and why didn’t any alert fire?
obs fail fullThe full Plan-§p6 synthetic-incident cycle: latency spike → error burst → retriever outage → recovery. ~26 min.
prompt The last half hour was rough and several different alerts fired. Reconstruct the timeline for me: each incident in order, what caused it, and whether they’re related.
obs fail <name> [qps]Optional second arg overrides the baseline traffic rate (default 40 qps). Bare obs fail lists all scenarios.
obs chaos [clear]Show which chaos knobs are live on model-proxy + retriever — or clear them (the panic button if a run died mid-drill).
obs fixes [clean]List the auto-fixer’s leftover workspaces (.artifacts/autofix) with sizes and pushed fix branches; clean deletes them. Each run auto-removes its big working clone on exit — only a ~1 MB origin.git with the fix branch remains. Pull a fix into the real repo with git fetch .artifacts/autofix/<run>/origin.git <branch>:<branch>.

Host processes (Phase 5 — not in compose)

# The web UI + agent-service run on the host so the Claude Agent SDK
# can use your Claude Code login. Easiest: one per terminal, after `obs up`:
obs agents     # agent-service :8093 — REQUIRED for RCA chat
obs web        # web control plane :3003

# Raw equivalents:
cd apps/agent-service; uv sync; uv run python -m agent_service
cd apps/web; bun run dev

Service addresses

Control plane http://localhost:3003 dev mode, no auth
Grafana http://localhost:3001 anonymous (Admin)
Marquez UI http://localhost:3002
Gateway API http://localhost:8080
dq-runner http://localhost:8091
Agent service http://localhost:8093 host process

Raw fallback (no wrapper)

# up — step 1 creates the shared networks, step 2 layers the rest
docker compose -f infra/compose.yml up -d --build
docker compose -f infra/compose.yml -f infra/compose.observability.yml -f infra/compose.lineage.yml up -d --build

# down (-v also wipes volumes)
docker compose -f infra/compose.yml -f infra/compose.observability.yml -f infra/compose.lineage.yml --profile load down

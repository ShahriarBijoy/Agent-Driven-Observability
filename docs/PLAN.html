AI Observability Lab

A self-contained, free-to-run learning lab where a small AI inference gateway becomes the subject of full-spectrum observation — logs, metrics, traces, lineage, and data quality — and Claude agents read that telemetry to diagnose, report, and act.

ModeLocal · Docker Compose
Cost$0 (FOSS only)
Cadence6 phases · ~10–14 days
LanguagesTypeScript · Python
OutcomeProduction-grade fluency

The lab is built around a single, opinionated insight: the most efficient way to learn production observability is to have a small but realistically broken system, instrument every layer of it, and then let AI agents reason about the resulting telemetry. The subject system is an AI inference gateway — a tiny LLM proxy with a RAG pipeline, embedding cache, rate limits, and synthetic faults. The observation system is the full Grafana LGTM stack plus OpenLineage / Marquez for data lineage. The intelligence layer is a Python service hosting five agents built on the Claude Agent SDK.

⚠ One thing to know before you start

As of June 15, 2026, Claude Agent SDK usage on subscription plans draws from a separate monthly Agent SDK credit, decoupled from your interactive Claude Code limits. The lab is designed for that: agents reason in fewer, longer turns rather than chatty per-event loops. Budget-friendly by design.

02 · OrientationArchitecture at a glance

Three planes, talking to each other through OpenTelemetry's OTLP protocol and OpenLineage's HTTP API. The subject system is on the left, the observation plane in the middle, and the intelligence plane on the right.

Three planes · one OTLP fabric · one repo

The three planes

The subject plane contains the AI inference gateway and its dependencies. Five services, two stateful stores. This is the thing you'd deploy in production; everything else is here to watch it.

The observation plane ingests two parallel streams: application telemetry over OTLP (going to Loki/Tempo/Mimir/Pyroscope via Grafana Alloy) and lineage events over HTTP (going to Marquez). Grafana federates the LGTM backends; Marquez has its own UI which the frontend embeds.

The intelligence plane hosts the agents. The agent service holds a small set of Python functions exposed to Claude as tools — LogQL queries, TraceQL queries, PromQL queries, Marquez lineage queries, Postgres reads, Grafana API writes, GitHub API writes, runbook reads. The web app is the human-in-the-loop surface: chat, approvals, incident inbox, embedded dashboards.

03 · OrientationStack and rationale

Every choice below was made to be free, well-documented in 2026, and worth learning. No proprietary SaaS. No cloud accounts required.

Subject system (TypeScript)

Runtime
Bun 1.2+ fast install, fast tests, native TS, single binary; what TanStack and Hono both target as first-class.
HTTP
Hono 4.x with @hono/zod-openapi tiny, fast, edge-portable, generates OpenAPI from Zod schemas you'd already write.
Validation
Zod 3.x single source of truth for runtime checks, OpenAPI, and shared client types.
ORM
Drizzle + pgvector type-safe SQL with the lowest abstraction tax in the JS ecosystem; pgvector is the easiest production-grade vector store.
Cache / RL
Redis 7 embedding cache + token bucket rate limiting.
Monorepo
Bun workspaces + Turborepo workspaces for hoisting, Turborepo for task graph + remote-cache discipline (even if you only use it locally).
Lint
Oxlint (OXC) Rust-based, ~2× faster than Biome and ~50–100× faster than ESLint; ~300 correctness/perf rules out of the box. No plugin system, but you won't need one here.
Format
Oxfmt (OXC, beta) same family as Oxlint, ~3× faster than Biome, supports JS/TS/JSON/CSS/MD/YAML/TOML. Beta but stable enough for a learning repo. If it ever bites, biome format is a drop-in fallback.
Test
Vitest + Bun:test Vitest for the rich ecosystem, Bun:test for service-level smoke tests where speed matters.

Frontend

Framework
TanStack Start (Vite + Nitro) file-based routing, server functions, streaming SSR, end-to-end type safety with TanStack Router.
Data
TanStack Query + TanStack Router loaders deduplicated caches, suspense-friendly, plays nicely with TanStack Start.
UI
Tailwind v4 + shadcn/ui copy-not-install components, plus the most ergonomic Tailwind release yet.
Auth
Better Auth (or dev token) drop-in, framework-agnostic, free; for local-only you can skip and use a fixed dev user.
Realtime
SSE for streams, WebSocket for approvals two complementary primitives, both cheap.

Observation

Standard
OpenTelemetry (SDKs in TS & Python) the only neutral standard; lets you swap any backend later without re-instrumenting.
Collector
Grafana Alloy the modern OTel Collector distribution from Grafana — its component model is the cleanest way to learn collector pipelines.
Logs
Loki label-indexed, content not indexed — forces you to learn good labeling discipline.
Traces
Tempo object-storage cheap; TraceQL is the most expressive trace query language in the open ecosystem.
Metrics
Mimir (Prometheus-compatible) scales further than vanilla Prometheus while speaking PromQL identically; you learn one query language for both.
Profiles
Pyroscope continuous profiling, integrated into Grafana as a 4th pillar.
Viz/Alerts
Grafana 12+ single pane, native correlation across signals via shared trace IDs.
Lineage
OpenLineage + Marquez open standard for data lineage; Marquez is its reference implementation and runs as one docker container.

Intelligence

Agent runtime
Claude Agent SDK (Python) same engine as Claude Code — built-in Bash/Read/Edit/Glob tools, MCP support, hooks for approval gates.
Service
FastAPI + httpx + pydantic a thin HTTP layer wrapping the SDK; the agents are the product.
Auth
Claude Code session (subscription) free under your subscription; switch to API key for production multi-user setups.

05 · OrientationThe vertical-slice DDD convention

Inside every TS service, code is organized by feature, not by layer. A feature owns its own domain model, its own ports, its own adapters, and its own HTTP handlers. The only cross-cutting concerns live in a small platform/ folder (telemetry init, config loading, error mapping, the Hono app bootstrap).

This convention is what makes the lab teach DDD rather than just talk about it: when you instrument a feature, you instrument the whole slice, not a layer. 

treeapps/gateway/
├── src/
│   ├── slices/
│   │   ├── inference/                  # the feature
│   │   │   ├── domain/
│   │   │   │   ├── prompt.ts           # value objects
│   │   │   │   ├── completion.ts
│   │   │   │   └── policies.ts         # business rules
│   │   │   ├── ports/
│   │   │   │   ├── embedder-client.ts  # interface
│   │   │   │   ├── retriever-client.ts
│   │   │   │   └── model-client.ts
│   │   │   ├── adapters/
│   │   │   │   ├── embedder-http.ts    # implementations
│   │   │   │   ├── retriever-http.ts
│   │   │   │   └── model-http.ts
│   │   │   ├── handlers/
│   │   │   │   ├── chat.ts             # POST /v1/chat
│   │   │   │   └── embeddings.ts

│   │   ├── config.ts                   # env loading via zod

│   │

├── otel.config.ts
└── package.json
A slice is the smallest piece of the system that emits useful telemetry on its own — its own spans, its own metrics, its own log labels. If you can't observe it independently, it isn't a slice yet.

Tasks

All knobs in env so the load-generator can crank them.
  6. apps/embedderHono service. POST /v1/embed takes text, returns a 384-dim vector. Use a deterministic hash-based fake embedder so the lab is reproducible. Wrap with Redis cache. Slices: embed/, cache/.

  8. apps/gatewayHono service. Each slice has its own slice.ts mounted in main.ts.
  9. apps/load-generatorA Bun script with weighted scenarios: happy path, repeat queries (cache-friendly), long prompts, abusive client (exceeds rate limit), broken client (malformed payload). Configurable QPS and run duration.
  10. Compose wiringAll services in infra/compose.yml, on the app network, with healthchecks. The load-generator is opt-in via profile.
  11. First test passPer-service Vitest specs for at least the inference orchestration and rate-limit logic. Smoke test: start compose, run scripts/smoke.sh, get a 200 from POST /v1/chat.
⌖ Subtlety worth noticing

The mock LLM is the keystone of the whole lab. Make its fault model rich: latency that looks like a real model under load, errors that cluster (a "bad minute" then recovery), occasional tail-latency stalls. Boring failure injection produces boring agents.

The OTel init pattern

Every service initializes the SDK in src/platform/telemetry.ts and imports it from main.ts before any other application code. The shared packages/telemetry exposes a single initTelemetry(service) function. 

ts · packages/telemetry/src/init.tsimport { NodeSDK } from '@opentelemetry/sdk-node';
import { Resource } from '@opentelemetry/resources';
import { ATTR_SERVICE_NAME, ATTR_SERVICE_VERSION }
  from '@opentelemetry/semantic-conventions';
import { getNodeAutoInstrumentations }
  from '@opentelemetry/auto-instrumentations-node';
import { OTLPTraceExporter  } from '@opentelemetry/exporter-trace-otlp-http';
import { OTLPMetricExporter } from '@opentelemetry/exporter-metrics-otlp-http';
import { OTLPLogExporter    } from '@opentelemetry/exporter-logs-otlp-http';

export function initTelemetry(service: { name: string; version: string }) {
  const sdk = new NodeSDK({
    resource: new Resource({
      [ATTR_SERVICE_NAME]: service.name,
      [ATTR_SERVICE_VERSION]: service.version,
    }),
    traceExporter: new OTLPTraceExporter(),
    metricReader: new PeriodicExportingMetricReader({
      exporter: new OTLPMetricExporter(),
    }),
    logRecordProcessor: new BatchLogRecordProcessor(new OTLPLogExporter()),
    instrumentations: [getNodeAutoInstrumentations()],
  });
  sdk.start();
}

Phase 03

Data observability

Treat the RAG pipeline as a data pipeline. Emit OpenLineage events for every inference, run scheduled data-quality checks, and surface freshness, drift, and quality as first-class SLIs alongside the app SLIs.

Effort 2 days
Risk medium · new mental model
Touches packages/lineage, apps/{gateway,embedder,retriever,dq-runner}, infra/marquez

What you'll learn

  • What data observability is and how it differs from app observability
  • The OpenLineage event model: jobs, runs, datasets, facets
  • How to think of an inference as a run of a job over datasets
  • Freshness, volume, distribution, schema — the four classic DQ checks
  • Why drift is the hardest of the four to define well
Mental model

In OpenLineage, every inference becomes a RunEvent of the job rag.inference. Its input datasets are vector_store.chunks and cache.embeddings; its output datasets are prompts.recent and completions.recent. Once you've mapped your system into that taxonomy, lineage falls out of the design.

Tasks

  1. Marquez in composeAdd Marquez + its Postgres in infra/compose.lineage.yml. Expose the API on a known port. Verify the UI loads with empty state.
  2. packages/lineageTS helpers to construct OpenLineage RunEvents with the right schema. Functions: start(run, job, inputs), complete(run, outputs, facets?), fail(run, err). POSTs to Marquez via fetch.
  3. Emit lineage from the gatewayWrap the inference slice with lineage emission. Each /v1/chat call is a run. Inputs: query embedding (from cache or computed) + retrieved chunks. Outputs: completion. Add custom facets for model, tenant, retrieval-score-stats.
  4. Emit lineage from the embedder & retrieverSub-runs that link back to the parent run via OL's parent-run facet. This is what lights up Marquez's lineage graph.
  5. apps/dq-runnerA Python FastAPI app with an APScheduler that runs DQ checks every 30 seconds. Reads from Postgres and Redis directly.
  6. DQ check — freshnessWhen was the last successful inference per tenant? Last vector-store update? Emit a metric dq_freshness_seconds labeled by dataset.
  7. DQ check — volumeRolling 1-minute count of inferences per tenant. Compare to a 1-hour baseline. Emit dq_volume_ratio and a violation event when ratio < 0.3 or > 3.0.
  8. DQ check — distribution / driftFor prompt length, retrieval relevance score, and completion token count, compute current 5-minute KS-distance from a 1-hour baseline. Emit dq_distribution_drift per signal. Violation threshold: KS > 0.25 (tune empirically).
  9. DQ check — schemaSample 100 recent responses, validate against the Zod schema in packages/contracts. Any failure → violation.
  10. DQ check — cache healthCache hit ratio per tenant. Anomalously low ratio → violation (signals either eviction or prompt-population shift).
  11. DQ violations tablePersist violations to Postgres: dq_violations(id, check, signal, severity, dataset, ts, payload jsonb). The web UI will query this in Phase 4; the agent in Phase 5.
  12. DQ dashboardGrafana dashboard "Data Quality" with: freshness per dataset, volume vs baseline, drift heatmap, cache hit ratio over time, violation rate. Embed Marquez's lineage graph as a linked panel (link out, not iframe).
  13. DQ alertsTwo Grafana alerts: any high-severity violation in last 5 minutes, prompt drift KS > 0.4 sustained 10 minutes. Same webhook contact point as Phase 2.
  14. ADR-002Document the lineage taxonomy: which entities are datasets, which are jobs, what facets you adopted. Future you will thank present you.

The tool layer

Before any agent runs, the agent service exposes a stable set of Python functions registered as Claude Agent SDK tools. Every agent shares this toolkit; only the system prompt and the allow-list differ.

Tool What it does
loki_query(logql, range) Returns log lines for a LogQL query over a time range
tempo_query(traceql, range) Returns matching trace summaries / a full trace by ID
mimir_query(promql, range, step) Returns instant or range-vector results
marquez_lineage(dataset, depth) Returns the lineage graph around a dataset
pg_select(sql, params) Read-only Postgres query (validated against an allow-list of tables)
grafana_create_dashboard(json) POSTs a dashboard JSON via Grafana HTTP API
gh_open_pr(branch, title, body, patch) Opens a PR in the subject repo via gh CLI
runbook_read(path) Reads a Markdown runbook from runbooks/
request_approval(prompt, payload) Blocks the agent until the human approves via the web UI
save_artifact(kind, content) Persists an artifact (Markdown, JSON) tied to the run

Tasks

  1. apps/agent-service skeletonFastAPI app with POST /runs, GET /runs/:id/stream (SSE), POST /runs/:id/approve. Python 3.12+, uv for dep management.
  2. Postgres schema for runsagent_runs(id, agent, status, started_at, ended_at, trigger, summary); agent_messages(run_id, ts, role, content); agent_tool_calls(run_id, ts, tool, input, output, duration_ms); agent_approvals(run_id, ts, prompt, decision, decided_at).
  3. Tool implementationsEach tool is a Python function with a clean signature, registered with the Claude Agent SDK via ClaudeAgentOptions(allowed_tools=[...]) or a custom tool plugin. Tools must validate inputs, log timings, and never raise — return structured errors.
  4. Agent: Incident ReporterEntrypoint: POST /webhook/grafana-alert. Spawns a session with the system prompt "you receive a Grafana alert payload and write a structured postmortem". Allowed tools: loki, tempo, mimir, marquez, pg_select, save_artifact. Output: a Markdown postmortem saved as an artifact and an incident row in Postgres.
  5. Agent: RCA AssistantEntrypoint: POST /chat (SSE stream). Long-lived session per conversation. Same read-only tool set as the reporter, plus marquez_lineage. The agent's job is to interactively answer "why is X happening" with real queries.
  6. Agent: Dashboard GeneratorEntrypoint: POST /generate-dashboard. Takes a natural-language brief. Allowed tools: mimir_query (to validate intended metrics exist), grafana_create_dashboard. Single turn, JSON in JSON out. No approval gate needed (dashboards are reversible).
  7. Agent: Auto-FixerEntrypoint: POST /auto-fix. Takes an error pattern (or is invoked by the reporter via tool chain). Allowed tools: Claude Code's built-in Read, Edit, Glob, Bash, plus gh_open_pr and request_approval. Required approval before the PR is opened. Runs in a contained workspace clone.
  8. Agent: Runbook ExecutorEntrypoint: POST /runbooks/:name/execute. Reads the runbook, parses headed steps, executes one at a time with request_approval between any two steps that mutate state. Allowed tools: Bash, pg_select, request_approval, save_artifact.
  9. Streaming + auditEvery tool call and message is written to agent_messages/agent_tool_calls in real time, and broadcast to the web UI via SSE/WebSocket.
  10. Self-observabilityInstrument the agent-service with OTel Python SDK. Each query() session is a parent span; each tool call a child span. This is critical — agents that don't appear in your traces are agents you can't trust.
  11. Wire Grafana alerts → reporterSet the Grafana alert contact-point URL to http://agent-service:<port>/webhook/grafana-alert. Verify a synthetic alert produces a postmortem.
  12. First synthetic incidentRun a chaos scenario (latency spike) for 3 minutes. Confirm the reporter writes a postmortem that names the slow span and the affected tenant. Iterate on the system prompt until the output is genuinely useful.
  13. First auto-fix dry runPlant a synthetic bug (e.g., a wrong default in rate-limit config) and feed the error pattern to the auto-fixer. Approve. Confirm a real PR opens against a local gh remote.
A good agent tool returns more than the LLM asked for. A great tool returns less. The art is knowing which it is — usually the answer is "less, but structured."

Phase 06

Chaos, SLOs, and the production-grade polish

Add synthetic incidents, SLOs with burn-rate alerts, cardinality control, tail-based trace sampling, profiling, and a "synthetic incident demo" script that exercises the entire system in one run.

Effort 2 days
Risk low–medium
Touches apps/load-generator, slo/, infra/{alloy,mimir,pyroscope}, scripts/

What you'll learn

  • SLO/SLI/error-budget thinking and how to express them in PromQL/recording rules
  • The multi-window multi-burn-rate alert pattern (the Google SRE one)
  • Cardinality budgets and how to enforce them at the collector layer
  • Tail-based trace sampling and what trade-offs you accept
  • Continuous profiling — when it's useful, when it's noise
  • Building a repeatable demo that teaches everything you just built

Tasks

  1. Chaos scenarios in the load-generatorScheduled scenarios on a clock: minute 0–6 happy path; minute 7 latency spike (3 minutes); minute 13 error burst (2 minutes); minute 19 dependency outage on retriever (2 minutes); minute 25 back to baseline. Knobs in YAML. Repeatable.
  2. SLO definitionsslo/gateway-availability.yaml: 99.5% of /v1/chat requests succeed (non-5xx). slo/gateway-latency.yaml: 95% of requests under 1.5s. slo/rag-quality.yaml: 90% of retrievals have top-1 score ≥ 0.6.
  3. Recording rulesFor each SLO, two recording rules: instantaneous SLI and rolling 28-day SLO. Commit to infra/mimir/rules/.
  4. Multi-window multi-burn-rate alertsPer SLO: a fast-burn alert (2% budget in 1h with both 5m and 1h windows on the burn rate) and a slow-burn alert (10% budget in 6h with both 30m and 6h windows). Wire to the reporter agent.
  5. Cardinality controlIn Alloy: drop or hash high-cardinality labels (e.g., user_id, request_id) before they reach Mimir. Document the rule and the rationale. Set a Mimir per-tenant series limit and verify the alert fires when you cross it.
  6. Tail-based samplingConfigure Alloy's tail_sampling processor: always sample errors, always sample traces over 1.5s, and 10% of the rest. Verify trace volume drops in Tempo storage while keeping interesting traces.
  7. PyroscopeAdd Pyroscope to compose. Use the OTel SDK's profiling support (or the Pyroscope Bun/Node integration) for the gateway. A new Grafana panel correlates a slow span to a CPU profile.
  8. Trace-log correlation hardeningConfirm every log line carries trace_id; in Grafana, configure Loki's "derived fields" to make trace_id a hyperlink to the corresponding Tempo trace, and vice versa from Tempo.
  9. RUM — browser REDAdd a "Frontend" row to the gateway dashboard with browser-reported request rate, error rate, and Largest Contentful Paint. Confirm that a slow API call in the gateway is visible in the browser trace.
  10. The Synthetic Incident demoscripts/demo-incident.sh: starts the lab, kicks off a chaos scenario, watches for the reporter's postmortem in the incident inbox, then runs the RCA assistant with a scripted follow-up question. Prints a transcript. This is the lab's "elevator pitch" — bring people to it.
  11. Documentation passWrite docs/learning-notes.md — for each phase, the three things you'd tell a colleague who's never built observability before. This is the artifact you keep.

B · ReferenceGlossary

Term What it means here
OTLP OpenTelemetry Protocol — the wire format for sending telemetry; HTTP/JSON, HTTP/protobuf, or gRPC.
Three pillars Logs, metrics, traces — the classic split. The exemplar is the seam between them.
Exemplar A trace ID stapled to a metric data point, so you can jump from "the p99 is high" to "this exact request".
RED Rate, Errors, Duration — the operator's first dashboard for any HTTP service.
USE Utilization, Saturation, Errors — the infrastructure equivalent. Less central in this lab.
SLI / SLO / SLA Indicator (measurement), Objective (target), Agreement (contract). This lab only deals with the first two.
Error budget The amount of "bad" allowed in your SLO window. Burn-rate alerts catch you spending it too fast.
Cardinality The number of distinct label combinations. Storage cost is roughly proportional. Hostile to user_id labels.
Tail sampling Decide whether to keep a trace after it ends. The collector buffers traces and applies rules. The alternative is head-based sampling (decide at the start).
Vertical slice A folder that contains everything a feature needs — domain types, ports, adapters, handlers, telemetry — instead of splitting by layer.
OpenLineage run A single execution of a job. Has start/complete events, input and output datasets, and arbitrary facets.
Marquez The reference OpenLineage backend. Stores runs, datasets, jobs; serves a graph UI.
Drift (data) A change in the statistical distribution of an input or output. KS-distance is one of many ways to detect it.
Approval gate A blocking checkpoint where an agent waits for a human OK. Implemented in this lab as a tool that suspends the run until the web UI posts a decision.
BFF Backend-for-frontend — a thin server layer dedicated to one client. Owns auth, payload shaping, fan-out. Lives inside apps/web.

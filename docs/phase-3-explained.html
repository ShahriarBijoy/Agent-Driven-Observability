AI Observability Lab · Phase 3

Data Observability, explained simply

Phase 2 asked “is the app healthy?” Phase 3 asks a harder question: “is the data it produces any good — and do we know where it came from?” Here's the whole idea, jargon unpacked, with a worked example.

01The problem we're solving

In Phase 2 we wired up the three pillars — logs, metrics, traces — so we could watch the app's vital signs: is it up, is it fast, is it throwing errors? That tells you the machine is running.

But our app is an AI/RAG pipeline — a machine whose whole job is to process data (find relevant chunks, write answers). And here's the trap: the machine can be perfectly healthy and still produce garbage. Every request returns 200 OK in 300 ms… while the answers quietly go stale, empty, weird, or stop coming entirely. Phase 2 would never notice — green lights all round.

Think of it like… A bottling factory. Phase 2 watches the machines: motors humming, no jams, belts at full speed — all green. Phase 3 inspects the bottles coming off the line: are they actually being filled? With the right liquid? At the normal rate? And if a bad batch shows up, can we trace it back to which ingredient and which step produced it? That second job — watching the product, not just the machine — is data observability.

So Phase 3 adds two new senses to the lab, and they answer two different questions:

🗺️

Data Lineage

“WHERE DID THIS COME FROM?”

A live map of how data flows: which inputs fed each request, which steps ran, and what came out. The family tree / supply-chain of every answer. Built with OpenLineage + Marquez.

🔬

Data Quality

“IS THIS DATA ANY GOOD?”

Scheduled health checks on the data itself — is it fresh, flowing at a normal rate, the right shape, the usual kind? Run every 30s by a small Python service we call the dq-runner.

02Idea A — Data Lineage (the map)

Lineage is just the paper trail of your data: this answer was produced by this run, which read these inputs and wrote those outputs. When something looks wrong, lineage lets you walk the chain backwards and find the culprit ingredient — exactly like a recall investigation in our bottling factory.

The vocabulary (only four words)

Word Means In our lab
Job A recipe — a named piece of work that runs over and over. rag.inference (one chat), rag.embed, rag.retrieve
Run One execution of a job, with a unique ID. One specific /v1/chat call, e.g. run 4bf92f35…
Dataset A named pile of data a run reads or writes. vector_store.chunks, cache.embeddings (in); prompts.recent, completions.recent (out)
Facet An extra sticky-note of detail attached to a run. retrieval scores, the model & tenant used, an error message on failure

The standard: OpenLineage

OpenLineage
The standard

A vendor-neutral standard for describing those jobs, runs, and datasets as little JSON “events.” If Phase 2's OpenTelemetry was the common language for app health, OpenLineage is the matching common language for data flow. Each run emits a START event when it begins and a COMPLETE (or FAIL) event when it ends.

The store + viewer: Marquez

Marquez
Lineage graph

The collector that receives those OpenLineage events, remembers them, and draws the graph you click around in. It is to lineage what Grafana is to metrics. You open it in a browser at localhost:3002.

Our package: @obs/lineage

@obs/lineage
Our emitter

A small package we wrote (the lineage twin of Phase 2's @obs/telemetry): it builds the START/COMPLETE/FAIL events and POSTs them to Marquez. It is best-effort — if Marquez is down, the error is swallowed and logged so a chat never fails just because lineage couldn't be recorded.

The parent / child trick — this is what draws the map One chat actually fans out to the embedder and retriever. We make the gateway's run the parent, and the embedder/retriever emit their own sub-runs (rag.embed, rag.retrieve) that point back to it via a parent facet. The link travels over HTTP headers (x-ol-parent-run-id & friends), exactly like trace propagation in Phase 2. Those parent→child links are the edges Marquez draws.

03Idea B — Data Quality (the inspector)

Lineage shows the shape of the flow. Data quality judges whether what's flowing is actually good. We do this with a tiny Python service, the dq-runner, that wakes up every 30 seconds, looks at the recent data, and grades it on five checks.

It can do this because the gateway now writes one row per successful chat to a new inferences table — prompt length, token counts, retrieval scores, cache flag, the response body, a timestamp. That table is the dq-runner's worktop: the recent data, laid out so it can be inspected.

The five checks

Check The question it answers Fires when…
Freshness “Is data still arriving, or did it go quiet?” nothing seen for > 120s medium
Volume “Are we getting the normal amount of traffic?” last-minute rate ÷ last-hour rate < 0.3 high or > 3.0 medium
Distribution
(drift)		 “Has the kind of data quietly changed?” KS distance > 0.25 medium, > 0.4 high
Schema “Is every response the right shape?” any of the last 100 responses fails the contract high
Cache health “Is the embedding cache actually helping?” hit ratio < 0.05 (with ≥20 samples) low
“Drift,” in plain words Imagine yesterday's prompts were mostly short, and today they're suddenly all long essays. Nothing errored — but the distribution shifted, and that often means something upstream changed. We measure it with the KS statistic (Kolmogorov–Smirnov): line up “normal” vs “now,” and it's the biggest gap between the two — a single number from 0 (identical) to 1 (completely different). Handy because it's already a ready-made 0–1 “how weird is it” score.

Where the results go — and why that's the clever bit

The dq-runner doesn't invent its own dashboard. It pushes its findings as dq_* metrics over OTLP to Alloy → Mimir — the exact same backend Phase 2 used for app metrics. So data-quality numbers sit right next to request-rate and latency in Grafana. That's what people mean by “data quality as first-class SLIs”: the freshness of your data is treated as seriously as the uptime of your service. When a check trips, it also writes a row to a dq_violations table — a permanent logbook of every problem found.

04The tools — who does what

Phase 3 introduces a few new players and re-uses the Phase 2 ones (shown in amber). Read it as two small teams.

OpenLineage
The standard

The common JSON language for describing data jobs/runs/datasets. Lineage team.

Marquez
Lineage store + UI

Collects the events and draws the clickable graph at localhost:3002. Lineage team.

@obs/lineage
Our emitter

Builds & ships lineage events from the TS services. Best-effort. Lineage team.

inferences
Postgres table

One row per successful chat — the record the dq-runner inspects. Quality team.

dq-runner
The inspector

Python service (FastAPI + a scheduler) that runs the five checks every 30s. Lives at localhost:8091. Quality team.

dq_violations
Postgres table

The logbook: one row each time a check trips (check, severity, details). Quality team.

Mimir + Grafana
reused from Phase 2

The same metrics store and dashboards. The dq_* metrics land here next to the app SLIs — no new viewer needed.

06A tiny worked example

Let's follow one chat — someone asks “what is pride and prejudice about?” — through both new flows.

① The lineage events. The gateway opens a run, the sub-services link to it, and it closes with facets:

# START — gateway begins run 4bf92f35…, declaring its inputs
{ "eventType": "START",
  "job":  { "namespace": "ai-observability-lab", "name": "rag.inference" },
  "run":  { "runId": "4bf92f35-77b3-…" },
  "inputs": [ "vector_store.chunks", "cache.embeddings" ] }

  # …two child sub-runs link back via a parent facet…
  rag.embed     (embedder)   parent → 4bf92f35…
  rag.retrieve  (retriever)  parent → 4bf92f35…

# COMPLETE — outputs declared, plus "sticky-note" facets
{ "eventType": "COMPLETE",
  "run": { "runId": "4bf92f35-77b3-…",
          "facets": { "retrievalStats": { count: 3, mean: 0.74, max: 0.81 },
                      "inference":      { model: "mock-llm", tenant: "acme" } } },
  "outputs": [ "prompts.recent", "completions.recent" ] }

Marquez stitches those into a clickable graph: chunks/cache → rag.inference → prompts.recent/completions.recent, with the embed & retrieve sub-runs hanging underneath.

② The row the gateway records (this is what makes prompts.recent / completions.recent real, and gives the inspector something to read):

-- one row in the `inferences` table, for this exact run
INSERT INTO inferences (run_id, tenant, model, prompt_chars, prompt_tokens,
    completion_tokens, retrieved_count, retrieval_score_mean, cache_hit, status)
VALUES ('4bf92f35-…', 'acme', 'mock-llm', 31, 42, 88, 3, 0.74, false, 'ok');

③ ~30s later the dq-runner sweeps the recent rows and emits its verdict as metrics — all healthy here:

# dq_* metrics → Mimir (these show up in the Data Quality dashboard)
dq_freshness_seconds{dataset="prompts.recent", tenant="acme"}   3      # fresh ✓
dq_volume_ratio{tenant="acme"}                            1.0    # normal ✓
dq_distribution_drift{signal="prompt_chars"}             0.08   # tiny shift ✓
dq_cache_hit_ratio{tenant="acme"}                         0.62   # cache helping ✓
Now break something Run the load-generator's broken scenario and watch successful chats collapse. Within one 30-second sweep, the volume check sees the rate crater and writes a row to the logbook — and the “any high-severity violation” alert fires: 
INSERT INTO dq_violations (check_name, signal, severity, dataset, payload)
VALUES ('volume', 'acme', 'high', 'prompts.recent',
        '{"ratio":0.12,"count_1m":2,"baseline_rate":17}');
That row — created automatically, within ~30s of the anomaly — is Phase 3 working.

07How we actually wired it in code

1

The gateway opens a lineage run per chat — a START up front, a COMPLETE (with facets) on success, a FAIL on error. This is the real gateway code: 

const runId = newRunId();
await deps.lineage.start({ runId, job: JOB_INFERENCE,
  inputs: [DS_VECTOR_STORE_CHUNKS, DS_CACHE_EMBEDDINGS] });
// …do the work…
await deps.lineage.complete({ runId, job: JOB_INFERENCE,
  outputs: [DS_PROMPTS_RECENT, DS_COMPLETIONS_RECENT],
  runFacets: { retrievalStats: stats, inference: { model, tenant } } });
2

Keep the run linked across services — the embed + retrieve calls run “inside” the parent, so the headers carry the parent run-id to the sub-services. This is what makes the graph connected (just like trace propagation in Phase 2): 

const parent = { runId, jobNamespace: …, jobName: "rag.inference" };
const { embedded, chunks } = await runWithParent(parent, () =>
  withSpan("rag.retrieve", async () => {
    const result = await deps.embedder.embed(input.prompt);   // → rag.embed sub-run
    return { embedded: result, chunks: await deps.retriever.retrieve(…) }; // → rag.retrieve
  }));
3

Record the inference — one best-effort row per successful chat into the inferences table. Best-effort means: wrapped in a try/catch so a DB hiccup never fails the user's chat. (Lineage emission is best-effort too.)

4

The dq-runner does the inspecting — a Python service (managed with uv) using FastAPI for its HTTP endpoints and APScheduler as the alarm clock that triggers a pass every 30s. Each pass reads inferences, runs the five checks (each guarded so one failure can't abort the others), pushes dq_* metrics over OTLP, and writes any dq_violations. The checks are plain, testable functions — e.g. drift is literally: 

# apps/dq-runner — the drift score is just the KS statistic
from scipy.stats import ks_2samp

def ks_drift(baseline, current, min_samples=20):
    if len(baseline) < min_samples or len(current) < min_samples:
        return None                                  # not enough data yet
    return float(ks_2samp(baseline, current).statistic)  # 0 = same, 1 = totally different
Why a separate Python service? Data-quality work is statistics — drift tests, sampling, distribution math. Python's scipy does this in one line, where TypeScript would be a slog. So the dq-runner lives in its own little Python world, but reports back into the same Grafana/Mimir everything else uses. (Full reasoning & thresholds: docs/adr/004-data-observability.md.)

08What you can see

Where What it shows you
Marquez (:3002) The lineage graph: the rag.inference job, its input/output dataset edges, and the rag.embed / rag.retrieve sub-runs — populating live as traffic flows.
Data Quality dashboard (Grafana, :3001) Freshness per dataset, volume ratio (banded 0.3–3.0), drift per signal (banded 0.25/0.4), cache-hit ratio, schema failures, and the violation rate by check & severity — plus a link-out button to Marquez.
/violations endpoint (:8091) curl localhost:8091/violations lists recent rows; curl -X POST localhost:8091/run triggers a check pass right now instead of waiting.
2 alert rules Auto-fire on (a) any high-severity violation in the last 5 min, or (b) prompt-length drift (KS) > 0.4 sustained for 10 min. Both page, alongside the Phase 2 alerts.

09See it yourself

Bring up the whole lab (subject + Phase 2 observability + Phase 3 lineage/quality), then pour in traffic:

# 1. Start everything (the broken/long scenarios will trip DQ violations)
bash scripts/dev-up.sh --build

# 2. Drive traffic so lineage + data-quality light up
GATEWAY_URL=http://localhost:8080 TARGET_QPS=120 DURATION_SECONDS=600 \
  bun --cwd apps/load-generator run start

Then:

  • Open Marquez → :3002 and watch the rag.inference graph fill in.
  • Open Grafana → :3001 → Data Quality to see freshness, volume, drift, and the violation rate.
  • Run curl localhost:8091/violations — a synthetic anomaly produces a row within ~30s. That little round-trip is Phase 3.

10One-screen recap

What Phase 3 did: taught the lab to watch the data, not just the app — where it came from, and whether it's any good.

  • Two new senses — Lineage (“where from?”) and Quality (“any good?”).
  • OpenLineage is the common language for data flow; Marquez (:3002) is its graph.
  • @obs/lineage emits parent/child runs so the graph is connected — best-effort, never breaks a chat.
  • The gateway records every ok chat to the inferences table — the inspector's worktop.
  • The dq-runner (Python, :8091) runs 5 checks every 30s: freshness, volume, drift (KS), schema, cache.
  • Results push as dq_* metrics into the same Mimir/Grafana as Phase 2 — “data quality as first-class SLIs” — and trips land in dq_violations.

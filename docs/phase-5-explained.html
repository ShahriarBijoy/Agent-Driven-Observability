AI Observability Lab · Phase 5

Claude Agents, explained simply

Phases 2–4 gave us a system that watches itself and a dashboard to look at it. But a human still has to sit there, read the charts, and figure out what to do. Phase 5 hires a small team of AI agents that read the same telemetry, reason about it, and — with your permission — act on it. Here's the whole idea, with no jargon left unexplained.

01The problem we're solving

By the end of Phase 4 the lab had a lot going for it. The app writes down everything it does (logs), counts and measures itself (metrics), records the journey of every request (traces), tracks where its data comes from (lineage), and checks whether that data is any good (data quality). And there's a nice web control plane — a single website — to look at all of it.

But notice who's still doing all the thinking: you. When something looks wrong, a person has to open the dashboards, run searches, read logs, follow a trace, connect the dots, write up what happened, and decide what to change. That's slow, it happens at 3 a.m., and it needs an expert who knows where everything lives.

Phase 5 asks: what if the telemetry had a reader? Not a chatbot that makes things up — a worker that actually goes and looks: runs the real searches, reads the real traces, and answers from what it finds. That worker is an agent.

Think of it like… Hiring on-call staff for a building full of sensors. Phases 2–4 installed the cameras, thermostats, and a security desk with screens. Phase 5 hires the people who actually watch the screens, walk down to investigate a beeping alarm, write the incident report, and — only after asking the manager — flip a switch to fix it.

02What is an "agent," really?

The word sounds fancy, but the recipe is short. An agent is just two things bolted together:

🧠

A model

THE BRAIN

Claude — the AI that can read, reason, and decide what to do next. On its own it can only talk. It can't see your logs or touch your system.

🔧

A set of tools

THE HANDS

Small, named actions we hand the model — "search the logs," "read a chart," "open a pull request." Each tool is a tiny program we wrote and trust.

Put them in a loop and you have an agent: the model looks at the goal, picks a tool, we run it and hand back the result, the model looks again, picks the next tool, and so on — until it has enough to answer or act. The model never touches your system directly; it can only ask for one of the tools we gave it. That's the whole safety story in one sentence: the agent can only do what's in its toolbox.

Think of it like… A detective who can't leave the office. They can't go anywhere themselves — but they can send a runner: "fetch me the door logs for last night," "pull the CCTV for camera 3." Each request comes back, they think, they send the next request. The runner (our tools) only knows how to do a few specific, safe errands. The detective (Claude) does the reasoning.

We built this with Anthropic's Claude Agent SDK — a toolkit for running this loop. One nice detail: it signs in using the Claude Code session already on your machine, so there's no separate API key to manage.

03The big picture, in one diagram

We built a new service called agent-service (written in Python). It sits between the website you click on and the telemetry the agents need to read. Here's the whole journey of a request:

# You, in the browser Control plane (web UI :3003) │ "Why is the gateway slow?" ▼ agent-service (:8090) ◀── the new Phase-5 backend │ ├─▶ Claude (the brain) decides: "I should check the metrics" │ │ │ ▼ picks a tool ├─▶ mimir_query ──▶ reads charts/numbers (Mimir) ├─▶ loki_query ──▶ searches the logs (Loki) ├─▶ tempo_query ──▶ follows one request (Tempo) │ │ │ ▼ results come back, Claude reasons again… ▼ An answer streams back to your screen, word by word, with the list of searches it ran shown alongside. Everything that happened is written down (audit trail) and also shows up as a traceable timeline (just like the app itself).

The clever part: the website didn't change. Phase 4 was built with a tidy "plug here later" seam, and Phase 5 simply plugged the real agents into it. The format of the messages flowing back to the browser stayed exactly the same.

04The toolkit — every tool, in plain words

All five agents share one toolbox. The only difference between agents is which tools each is allowed to use and what job we tell it to do. Below is every tool. The coloured pills tell you whether a tool only reads (safe, can't break anything), changes things, or pauses for your approval.

Tools that read the telemetry

loki_query
reads

Searches the logs. Logs are the diary lines every service writes — "received a request from acme", "error: timed out talking to model-proxy". The agent gives it a search (in a small query language called LogQL, e.g. "show me lines from the gateway that contain the word error") and gets back the matching lines. This is how an agent finds error messages and specific events.

tempo_query
reads

Follows one request's whole journey. A trace is the timeline of a single request as it hops from service to service (gateway → retriever → model-proxy), showing how long each hop took. Each hop is called a span. The agent uses this to find the slow step — "the request waited 30 seconds inside model-proxy." It can search for slow traces, or pull up one trace by its ID.

mimir_query
reads

Reads the numbers and charts. Metrics are counts and measurements over time: requests per second, the share of requests that failed, the "95th-percentile" response time (the slow end of normal). The query language here is PromQL. The agent uses this to measure a problem — "errors are at 5%, which is above the 2% limit."

marquez_lineage
reads

Maps where data comes from. Lineage is the family tree of a dataset: "this table was built by that job, from those inputs." The agent uses it to understand what feeds what — useful when a bad output might trace back to a bad input upstream.

pg_select
reads

Looks things up in the database — read-only. The lab stores rows in a Postgres database (past requests, data-quality violations, the agents' own history). This tool can run only "SELECT" queries (the read kind) and only on a short list of safe tables. It is physically unable to change or delete anything — we check every query before running it and refuse anything that isn't a plain read.

Tools that do something in the world

grafana_create_dashboard
changes things

Builds a new chart page in Grafana. Grafana is the dashboard app. The agent hands it a description of the panels it wants, and Grafana saves a real, openable dashboard. We treat this as safe-enough to not need approval, because a dashboard is just a view — you can always delete it, it can't hurt the running system.

runbook_read
reads

Opens a printed instruction sheet. A runbook is a checklist document for handling a specific problem ("gateway error rate is high — here's what to check and how to fix it"). They live as plain text files in the runbooks/ folder. This tool reads one so the agent knows the steps. It's locked to that folder — it can't be tricked into reading files elsewhere.

gh_open_pr
changes things needs approval

Proposes a code change as a pull request. A pull request (PR) is a suggested edit to the code that a human reviews before it's accepted — nothing is changed on the live system, it's just a proposal on a side branch. The agent only ever calls this after you've approved. In the lab it pushes to a local practice copy, so it's a safe dry run.

Tools for the human-in-the-loop and record-keeping

request_approval
pauses for you

Asks your permission, and waits. This is the safety valve. When an agent is about to do something that changes the system, it calls this tool with a one-sentence description ("restart the gateway?"). The agent freezes — it literally stops and does nothing — until you click Approve or Deny on the run's page. If you deny, it skips the action.

save_artifact
saves a file

Saves its work as a file you can read later. When an agent produces something worth keeping — a written-up incident report, a summary — it stores it as an artifact attached to that run, so you can open it from the web UI.

Built-in tools (only the bolder agents get these)

Two agents — the runbook executor and the auto-fixer — also get a few of Claude's standard built-in tools so they can work with the system and the code directly:

Read · Glob
reads

Open a file; find files by name. How the auto-fixer locates the buggy file and reads it before changing anything.

Edit
changes a file

Change a few lines in a file. The auto-fixer makes the smallest correct edit — and only inside a safe copy of the code, never the real one (more on that in §10).

Bash
runs a command

Run a terminal command. Used to run a check or carry out a runbook step. State-changing commands are gated behind request_approval.

05The five agents

Same toolbox, same brain — five different jobs. Each agent is really just a job description (what we tell Claude its role is) plus an allowed tools list (which of the tools above it may use).

① RCA Assistant read-only

The detective you can chat with. "RCA" means root-cause analysis — figuring out the real reason something is happening. You type a question on the /agents page ("which tenant is sending the most traffic, and what's the current slow-end latency?") and it answers by actually running the searches, with the list of searches shown live next to the chat. It can only read, so it can never break anything.

We watched it do this When asked about the gateway, it first guessed a metric name, found nothing, then queried to discover the real names, and only then answered — "≈3.1 requests/second, zero errors, but a recurring warning about lineage." That self-correction is the whole point: it didn't guess, it looked.

② Incident Reporter read-only

The night-shift writer. When an alarm fires in Grafana (say, "gateway errors above 2%"), Grafana pings the agent-service, which wakes this agent up. It investigates with the read tools, then writes a postmortem — a tidy report of what happened, who was affected, the likely cause (naming the exact slow step), and what to do next — and files it in the /incidents inbox. In our test it produced an 8-page report that correctly fingered the slow model-proxy step.

③ Dashboard Generator makes a dashboard

The chart-maker. You describe a dashboard in one sentence ("gateway health: request rate, slow-end latency, and error share over time") and it builds a real Grafana dashboard you can open. It first checks that the numbers it wants actually exist, then creates the page. No approval needed — a dashboard is just a view.

④ Runbook Executor approval at each risky step

The careful operator. Give it the name of a runbook and it reads the checklist and works through it one step at a time. The read-only steps it just does — but before any step that changes the system (restart a service, write a file) it stops and asks you. In our test it ran the safe checks, paused at the "write a snapshot file" step, and when we clicked Deny, it skipped it and finished — the file was never written. That's the guarantee working.

⑤ Auto-Fixer approval before the pull request

The junior engineer. The boldest agent, built last and most carefully. Give it a description of a bug and it finds the file, makes the smallest fix, double-checks it, and opens a pull request — but only after you approve, and only inside a throwaway copy of the code (so the real project is never touched). In our test we planted a deliberately-wrong setting; it found it, changed it to the right value, paused for approval, and — once we clicked Approve in the web UI — opened the PR.

06A run, step by step

Here's exactly what happens when you ask the RCA assistant a question:

1

You ask. You type a question in the web UI and hit send. The website forwards it to the agent-service.

2

A "run" is created. The agent-service opens a new record (a run) to track everything about to happen, and starts streaming updates back to your screen.

3

Claude picks a tool. It reads your question and decides, say, mimir_query is the right move. The agent-service runs that tool against the real metrics and hands the result back.

4

It loops. Claude looks at what came back and picks the next tool — maybe a log search, maybe a follow-up metric. Each tool call appears live in the "tool timeline" panel so you can watch it work.

5

It answers. Once it has enough, it writes the answer, which streams to your screen word by word. The whole thing — every message, every tool call, the final answer — is saved to the database.

07The human in the loop: approvals

The most important safety feature is also the simplest: for anything risky, the agent stops and asks. Technically, when an agent calls request_approval, its whole run freezes mid-thought and parks itself, waiting. Nothing else happens. The run's status changes to "awaiting approval," and a card with Approve / Deny buttons appears on its page.

When you click a button, your decision travels back to the frozen agent, which un-freezes and either continues (Approve) or skips the action and explains why (Deny). You are always the one who pulls the trigger on anything that changes the system.

Think of it like… A trainee who's been told: "do all the looking and prep work yourself, but never push the big red button without a manager standing next to you." They'll get everything ready, then wait at the button until you say go.

08Remembering everything: the audit trail

Trust needs a paper trail. Every single thing an agent does is written to the database as it happens — not as an afterthought. There are five record books (database tables):

Record book What it holds
agent_runs One row per run: which agent, when, and how it ended.
agent_messages Every message — what you said, what the agent said back.
agent_tool_calls Every tool the agent used, with its inputs, results, and how long it took.
agent_approvals Every time it asked permission, and what you decided.
agent_artifacts Every file it produced (postmortems, summaries).

Because it's all in the database, the web UI can show you any past run in full — and nothing the agent did is hidden. There's a sixth table, incidents, that powers the incident inbox.

09Watching the watchers

Here's a lovely twist. This whole lab is about observability — being able to see inside a running system. So we pointed that same lens at the agents themselves. Every agent run shows up in Tempo (the trace viewer) exactly like a normal request: one big timeline for the run, with a smaller timeline nested inside for each tool it used.

agent.incident-reporter ← the whole run (one trace) ├─ tool.mimir_query ← each tool call is a nested step ├─ tool.loki_query ├─ tool.tempo_query └─ tool.save_artifact

We verified this live: a real incident-reporter run appeared in Tempo as one parent timeline with 22 nested tool steps. The principle: an agent you can't see in your traces is an agent you can't trust.

10Keeping it safe

Several layers, each easy to state:

  • The agent can only use its toolbox. It cannot run arbitrary actions — only the specific, reviewed tools we handed it, and only the ones on its allow-list.
  • Reading is separated from changing. Most agents (RCA, incident reporter) get only read tools. They are incapable of altering anything.
  • The database tool can't write. pg_select checks every query and refuses anything that isn't a plain read of a known-safe table.
  • Risky actions stop for you. Restarting a service or opening a pull request always waits for your Approve.
  • The auto-fixer works on a copy. It edits a throwaway clone of the code with its own private practice "remote," so the real project is never touched during a dry run.
  • We fixed a real flaw during the build. An automated security review noticed that the "save a file" tool trusted a filename the AI chose — which could, in theory, write outside its folder. We tightened it to only allow plain, safe filenames, and added a test so it can't regress.
One practical gotcha The agent-service runs as a normal program on your computer (so it can use your Claude sign-in). On Windows, the word localhost sometimes points at the "new-style" network address while the service listens on the "classic" one, and they don't meet. The fix was simply to tell the website to reach the service at 127.0.0.1 (the classic address) instead of localhost. Mentioned here in case you ever see a "run not found" and wonder why.

11What we proved

Phase 5 isn't "it should work" — we ran each agent against the live system and watched it in the browser. Every one of the plan's success criteria came back green:

The goal What we saw
An alert produces a postmortem in the inbox. A test "gateway latency" alert → an 8-page report naming the slow model-proxy step and the affected tenant, in /incidents.
Ask the RCA assistant "why is X happening?" and get a real, query-backed answer. It ran 10+ real metric/log searches (shown in the timeline) and answered with actual numbers.
The dashboard generator creates a real dashboard from a sentence. A "Gateway Health" dashboard appeared in Grafana, confirmed via Grafana's own search.
The auto-fixer opens a PR for a planted bug, with an approval gate the UI honors. We planted a wrong setting; it fixed it, paused, and opened the PR after we clicked Approve in the web UI.
The runbook executor walks a runbook with per-step approvals. It paused at the risky step; we clicked Deny and it correctly skipped it (file never written).
Every agent run is a trace in Tempo, tool calls nested inside. Confirmed: parent run timeline with one nested step per tool call.

12Try it yourself

Three terminals. The lab's backends run in Docker; the agent-service and website run directly on your machine. 

# 1) Bring the whole lab up in Docker (databases, Grafana, the app, traffic):
bash scripts/dev-up.sh --build

# 2) Start the agent-service (it uses your local Claude sign-in):
cd apps/agent-service
uv sync
uv run python -m agent_service        # listens on :8090

# 3) Start the control plane (the website):
bun --cwd apps/web run dev            # open http://localhost:3003

Then, in the browser:

  • /agents — chat with the RCA assistant and watch the tool timeline fill in.
  • /incidents — read postmortems. Make one by firing a pretend alert: 
    curl -X POST localhost:8090/webhook/grafana-alert \
  -H 'content-type: application/json' \
  -d '{"status":"firing","alerts":[{"status":"firing",
       "labels":{"alertname":"Gateway 5xx rate > 2%","severity":"page"},
       "annotations":{"summary":"gateway 5xx above 2%"}}]}'
  • Make a dashboard: 
    curl -X POST localhost:8090/generate-dashboard \
  -H 'content-type: application/json' \
  -d '{"brief":"gateway health: request rate, slow-end latency, error share"}'
  • Run a runbook (then Approve/Deny on the run's page): 
    curl -X POST localhost:8090/runbooks/snapshot-agent-audit.md/execute
The takeaway Phase 5 turns the lab from a system you watch into a system that has helpers who watch it for you — helpers that only ever look unless you say otherwise, write down everything they do, and show up in the very telemetry they're reading. That last bit is the whole spirit of the project: even the watchers are watched.
